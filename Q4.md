# Why does BitNet use separate kernel implementations for ARM (TL1) and x86 (TL2) instead of a unified kernel, and what hardware differences require this split?

## **1 . Introduction**

BitNet maintains two separate kernel implementations:

* **TL1 → optimized for ARM architectures**  
* **TL2 → optimized for x86 architectures**

These implementations differ in:

* matrix layouts,  
* block sizes,  
* preprocessing pipelines.

Instead of unifying them into a single portable kernel, BitNet deliberately keeps them separate to **maximize performance by leveraging hardware-specific optimizations**.

### Direct Answer

BitNet does not unify kernels into a single implementation because ARM and x86 architectures require fundamentally different data layouts, vectorization strategies, and execution patterns.

Although conditional compilation is used to select between TL1 and TL2, a single portable kernel would reduce performance and limit hardware-specific optimizations.

The split is necessary due to differences in SIMD instruction sets, vector widths, memory alignment, and cache behavior.

## **3 . Why Not Use a Single Unified Kernel?**
The differences between architectures are not limited to instruction sets but extend to overall computation structure and data layout.

---

### **3.1 Performance Optimization**

A unified kernel would require abstraction layers or conditional logic, which would:

* reduce the ability to use low-level hardware instructions efficiently,  
* introduce runtime overhead,  
* limit compiler optimizations.

Separate kernels allow each architecture to be fully optimized for maximum performance.

---

### **3.2 Instruction Set Incompatibility**

ARM and x86 CPUs use different instruction sets:

* ARM → NEON SIMD instructions  
* x86 → AVX/AVX2/AVX-512 SIMD instructions

These differ in:

* supported operations,  
* execution behavior,  
* instruction format.

This makes it impractical to implement a single kernel that efficiently supports both.

---

### **3.3 Vector Width Differences**

Vector width determines how much data can be processed in one instruction:

* x86 (AVX2) → 256-bit vectors  
* ARM (NEON) → 128-bit vectors

This affects:

* loop design,  
* batching strategy,  
* throughput.

A unified kernel would not be able to fully utilize both architectures efficiently.

---

### **3.4 Memory Layout and Data Packing**

Different architectures prefer different:

* memory alignment strategies,  
* data layouts,  
* packing formats.

Efficient memory access is critical for performance.  
Separate kernels allow preprocessing pipelines to be tailored for each architecture.

---

### **3.5 Cache Behavior and Block Size Optimization**

ARM and x86 differ in:

* cache size and hierarchy,  
* memory access patterns.

This affects:

* optimal block sizes,  
* tiling strategies,  
* loop ordering.

Separate kernels allow these parameters to be tuned for each architecture.

---

## **4 . Micro-Architecture Differences Necessitating the Split**



### **4.1 SIMD Instruction Set**

* ARM uses NEON SIMD instructions  
* x86 uses AVX/AVX2/AVX-512 instructions

These determine how vector operations are executed at the hardware level.

---

### **4.2 Vector Register Width**

* ARM → typically 128-bit  
* x86 → 256-bit (or higher)

This directly impacts how many elements can be processed at once.

---

### **4.3 Data Layout and Alignment Requirements**

Each architecture benefits from different:

* data layouts,  
* alignment rules,  
* tensor storage formats.

---

### **4.4 Cache Architecture**

Differences in:

* cache size,  
* cache hierarchy,  
* memory bandwidth

affect how data should be accessed and processed efficiently .

---

### **4.5 Hardware Design Goals**

* ARM → optimized for power efficiency and mobile/embedded use  
* x86 → optimized for high performance and throughput

These design differences lead to different optimization strategies.

---

## **5 . Summary**

* BitNet uses separate kernels (TL1 for ARM, TL2 for x86) to fully utilize hardware-specific optimizations.  
* Conditional compilation is already used, but it only selects between implementations rather than unifying them.  
* A unified kernel would reduce performance due to abstraction and inability to exploit architecture-specific features.  
* Differences in SIMD instructions, vector width, memory layout, and cache behavior necessitate separate implementations.  
* These differences extend beyond instructions to full computation structure and data layout.

---

## **6 . Final Insight**

BitNet’s design reflects an important systems principle:

**“Performance-critical systems should use hardware-specific optimizations rather than generic abstractions.”**
