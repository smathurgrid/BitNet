## **1 . Introduction**

BitNet’s MAD backend implements two execution modes:

* **Non-parallel mode** → processes data in fixed groups of 4 rows  
* **ACT _PARALLEL mode** → processes rows independently in parallel

A key constraint in the non-parallel implementation is:

nrow % 4  == 0

This ensures that the number of rows is divisible by 4 .

## **3 . Why Non-Parallel Mode Requires nrow % 4  == 0**

From the code structure:

* computation is performed in fixed blocks of 4 rows,  
* packing combines exactly 4 rows into 1 byte,  
* indexing logic assumes complete groups.

From a theoretical perspective:

* **block-based processing improves efficiency** by reducing control overhead and enabling predictable memory access patterns,  
* however, it requires **strict data alignment**, as operations are designed for fixed-size groups,  
* such designs trade flexibility for performance and simplicity.

**Conclusion:**  
Non-parallel mode requires `nrow % 4 == 0` because the entire computation pipeline is built around fixed 4-row grouping and assumes complete blocks.

---

## **4 . What Breaks Without This Constraint**

---

### **4.1 Incomplete Row Groups**

If rows are not divisible by 4:

* the last group contains fewer than 4 rows,  
* variables corresponding to missing rows (e.g., r2, r3) become invalid.

From theory:

* block-based algorithms require **complete data units** to maintain correctness,  
* partial groups introduce undefined states.

**Conclusion:**  
Incomplete groups cannot be processed correctly.

---

### **4.2 Invalid Memory Access**

From indexing:

q8 [r3  * n _per _row  + col ]

If `r3` exceeds valid rows:

* memory outside valid bounds is accessed.

From theory:

* low-level memory operations assume valid indexing,  
* accessing out-of-bound memory leads to undefined behavior or crashes.

**Conclusion:**  
This results in unsafe memory access and potential runtime failures.

---

### **4.3 Incorrect Bit Packing**

Packing assumes:

 [q0 q1 q2 q3 ]

If fewer rows exist:

 [q0 q1 ? ? ]

From theory:

* bit packing requires **complete and consistent data units**,  
* missing values lead to incorrect bit representation.

**Conclusion:**  
Produces corrupted packed values that no longer represent valid data.

---

### **4.4 Incorrect Downstream Computation**

Later stages (e.g., decoding, dot product) assume:

* exactly 4 valid values per byte.

If packing is incorrect:

* decoding extracts invalid values,  
* computations propagate these errors.

From theory:

* errors in early stages of a pipeline propagate and amplify in later stages.

**Conclusion:**  
Final computation results become incorrect.

---

## **5 . Why Parallel Mode Does Not Require This**

In ACT _PARALLEL mode:

* rows are processed independently,  
* computation does not depend on fixed-size groups,  
* flexible batching is used.

From theory:

* parallel processing models are designed to handle **irregular data sizes**,  
* they prioritize scalability and flexibility over strict structural constraints.

**Conclusion:**  
Parallel mode avoids the divisibility constraint because it does not rely on fixed grouping.

---

## **6 . Why This Design is Used**

BitNet combines two complementary approaches:

* **Non-parallel mode** → optimized for structured, block-based computation  
* **Parallel mode** → optimized for flexibility and scalability

This ensures:

* efficient memory usage,  
* high-performance computation,  
* adaptability across different workloads and hardware conditions.

---

## **7 . Final Insight**

This design reflects an important systems principle:

**“Block-based optimizations improve efficiency but require strict data alignment, whereas parallel execution provides flexibility at the cost of more complex handling.”**

---

## **Final Conclusion**

* Non-parallel mode requires `nrow % 4 == 0` because it relies on fixed 4-row grouping, structured packing, and aligned memory access.  
* Violating this constraint leads to incomplete groups, invalid memory access, corrupted data representation, and incorrect computation results.  
* Parallel mode removes this constraint by processing rows independently, allowing flexibility for arbitrary input sizes.

**Thus, the constraint is essential for correctness and efficiency in block-based processing, while parallel execution provides a flexible alternative without such restrictions.**
