# Why does BitNet use int8 activations and int2 weights instead of using int2 for both?



## **1 . Introduction**

BitNet’s GPU implementation performs matrix multiplication using:

* **int8 activations**  
* **int2 weights**  
* **bfloat16 output**

Instead of using int2 for both activations and weights, BitNet adopts a **mixed-precision strategy** to balance computational accuracy and memory efficiency.

### Direct Answer

BitNet uses int8 for activations and int2 for weights because activations are dynamic and require higher precision, while weights are static and can be aggressively compressed.

Using int2 for activations would cause severe information loss and large errors in computation, whereas int8 preserves sufficient dynamic range.

Thus, mixed precision (int8 × int2) balances accuracy and memory efficiency.

## **3 . Why Activations Use int8**

Activations are:

* dynamically generated during inference,  
* dependent on input data,  
* highly sensitive to precision loss.

This implies:

* activations require runtime scaling,  
* and their range varies across inputs.

Using int8:

* provides 256 possible values,  
* preserves sufficient information during computation.

**Conclusion:**  
Activations require int8 precision to maintain accuracy during computation.
Errors in activations propagate through every multiplication in the dot product (Σ wᵢ × xᵢ), making their precision significantly more critical than weights.

---

## **4 . Why Weights Use int2**

Weights are:

* fixed after training,  
* optimized to tolerate quantization.

This implies:

* weights can be heavily compressed,  
* memory usage can be minimized.

Since weights do not change during inference:

* aggressive quantization (int2) is feasible.

**Conclusion:**  
Weights can be stored in int2 to maximize memory efficiency with minimal accuracy impact.

---

## **5 . Why Not Use int2 for Both**

Using int2 for activations would mean:

* only 4 possible values,  
* extremely limited representation.

Consequences:

* poor representation of dynamic values,  
* inaccurate intermediate computations,  
* significant degradation in model performance.

Unlike weights:

* activations require higher precision due to their variability.

**Conclusion:**  
Using int2 for activations would severely degrade accuracy, making it impractical.

---

## **6 . Why Mixed Precision (int8 × int2) is Optimal**

BitNet’s design balances:

* memory efficiency → int2 weights  
* computational accuracy → int8 activations

This ensures:

* reduced model size,  
* efficient computation,  
* minimal accuracy loss.

**Conclusion:**  
Mixed precision provides the best trade-off between efficiency and accuracy.

---

## **7 . Why Output is bfloat16**

After low-precision computation:

* rounding and quantization errors accumulate.

Using bfloat16:

* provides higher numerical range than integer formats,  
* helps stabilize final outputs,  
* maintains output quality.

**Conclusion:**  
bfloat16 output ensures stable and reliable final results.

---

## **8 . Final Insight**

BitNet’s design reflects a key principle:

**“Different components require different precision levels based on their role in computation.”**

---

## **Final Summary**

* Activations are dynamic and require higher precision → use int8.  
* Weights are static and can be compressed → use int2.  
* Using int2 for activations would cause severe information loss and large computation errors.  
* Mixed precision (int8 × int2) balances accuracy and memory efficiency.  
* bfloat16 output ensures stability after low-precision computation.
